[IF YOU'RE NEW HERE]

If you've read up to this point and aren't quite sure what "Everytale" is, allow me to clarify. Everytale is an online Undertale fangame I'm currently developing. The game combines social game elements (i.e. VRChat) with Undertale style turn-based combat for PvE and PvP. Players can hang out, role-play, chat, and battle in an Undertale/Deltarune style world, together! The game will also feature built-in mod tools so players can import/create their own custom characters to play and fight as, all completely cross-platform. If you want more details on the game, you can read the game design document here.

[THE FINAL PROTOTYPE]

I am reffering to the previously mentioned "Dynamic Resources Test" as "the final prototype" because I will not be making any more prototype projects at this point. The game is now entering pre-alpha, meaning work on the actual project has begun (elaborated on more in [PRODUCTION]). All programming I'll do from this point forward will be coding the version of the game you will eventually get to play. In the meantime, everything I've shown off for Everytale up to this point has been theoretical, so why don't we change that! I believe showing off these prototyped systems are a good way to show my intentions for the final product, even if they're neccesarily scrappy.


Please keep in mind that everything you're about to see are prototypes. The purpose of prototypes, in my mind, is to rapidly test the implementation of systems before adding said systems to the real project. This allows me to find any potential problems with my implementation early on and avoid technical debt. Everything shown in this section is purposefully basic, unfinished, unpolished, and lacking in UI. These prototypes are not meant to represent the quality of the final game, which will be far cleaner and far more polished.


Without further ado...



Here's how the prototypes are organized in-editor! There are a variety of points of interests here, but the main thing I would like to point your attention towards is panel on the right side of the screen. There, you can see the player object has a variable called "costumes," which holds an array of costume resources. Every costume resource holds:


  • A spriteframes object that holds the image data that gets used in animations.
  • The frame change delay, which effectively controls how quick the costume's walk animations are.
  • The walking animations frame arrays, which tells Godot how long each walk cycle should be and which sprites from the spriteframes object should called, and at what time.
  • The sprite offset, which determines how far the sprite object should be from the actual center of the player object.
  • A variety variables that control the collision capsule.

The point of all this? Those of you who have read the game design document will remeber the costume system, where the idea is that a single "character" can have multiple "costumes" that they can switch between on the fly. I have these settings set up in the editor as this is a prototype, so spending time designing UI would be a waste, but in the playable version of Everytale these will be some of the parameters players will use to create custom characters in game.


But enough about the editor. What about in-game?



Bam! As you can see, the player in this case is switching between a variety of costumes on the fly. Costumes can even be changed whilst moving, and everything still works!


It should be noted that this is technically an improper use of the costume system. In the real game, the costumes array is intended to be held within a "character" resource (which it is not here), and each costume is supposed to relate to the character. For example, the character resource "Ralsei" might have three costumes: one with his hat on, one with it off, and one from when he wears a suit in Deltarune Chapter 2. The reason I'm choosing to show it off like this with completely unrelated characters is because I want to emphasize how dynamic this system is intended to be. Each of the three costumes shown off have completely different sprites, frame change delays, collision capsule sizes, etc. Costumes can be completely different from each other, and they will still work. This will be huge for player expression, and allow people to create some incredibly creative and dynamic characters.


Speaking of things that are dynamic, did you notice the video's aspect ratio? It's not in 4:3.





Something I never really liked in Undertale, and equally disliked that fangames followed suit with, is that they're in 4:3. This doesn't take full advantage of the screen space and creates two ugly black bars on each side of your screen. Some fanagmes try to remedy this by having some nice art in place of where the black bars would normally be, but that only solves the second problem. Because Godot has fantastic UI tools, I've decided Everytale will be designed with dynamic aspect ratio support. This means the base aspect ratio is 4:3, but if the window is resized to be an aspect ratio wider than that, the game will keep the same height and increase the number of pixels visible to the left and right. This means that Everytale will be able to fill almost any screen size, which will be especially important for making the mobile version feel good to play. Most critically, however, is that the extra screen space is a great place for the chat to sit!



And with that, you've seen the majority of what I prototyped before I began working on the real deal! Before we talk about the state of the main project, there is one last thing I'd like to show you. I'm putting this at the end because I don't really know how to transition into this, as it's not that interesting and disconnected from everything else. While for the most part I knew how I was going to design Everytale's battle system, I wasn't sure how to make the box the player dodges in dynamic. [PRODUCTION]

After I was satisfied with all the prototypes I had created, and as I began to notice it was now approaching one month since I posted the game design document, I decided it was finally time to get the real project set up in Godot, and set up the repository in GitHub. It might sound silly to dedicate a section to something like this, but it really is both a big step and something I felt I needed to be careful about. It's a big step because setting all this up marks the beginning of a new phase of production, and I needed to be careful because the way I structure the project early on sets precedent for the rest of the game's development, so it's important that said precedent is a healthy one.


When I set up the project, one of the most critical decisions I had to make was how I was going to structure the file system. Keeping files organized matters in a project like this due to the sheer number of files that will need to be stored. In Dynamic Resources Test, along with every other prototype that came before it, I could reasonably get away with storing files willy-nilly because they were all so small I knew I wouldn't have to worry about finding files later. This is not true for Everytale. But I don't have much experience working with large codebases, so I knew I would need to put extra effort into ensuring the file system stayed navigatable even at a large scale. In the end I would design it something like this:



Below the root there are three main folders:


  • Assets, which holds art, sound effects, music, fonts, etc.
  • Objects, which holds scene files and script files.
  • Resources (currently absent), which will hold both Godot and custom resource files.

Then, each child folder groups categories of files together, getting more specific the further you go down. Organizing the file system this way can make creating new files a pain sometimes, but ultimately avoids technical debt and significantly improves project management.


You might notice the term "pre-alpha" in the screenshot above. I'm happy to announce that Everytale is now offically in the pre-alpha stage of development! Before, the game was in a combination of the "planning" and "pre-production" stages of development (both simultaneously because I'm only one person). Now that the game is being built, we are in the pre-alpha stage of production. The term "pre-alpha" has a fairly loose definition, and is usually used to describe the stage between the pre-production stage and the alpha stage. A game goes into alpha when it becomes feature complete, but isn't content complete yet. Since we just got started, we are a very long way from the alpha stage. You can expect Everytale to be in pre-alpha for most of its development life.





[GOING FORWARD]

You are now fully caught up on where we are.


The only question left is where do we go from here?


The first order of business for Everytale in terms of development will be building the "cosmetic" system (which I'm thinking of renaming to the "character" system) from the game design document. This will be the modded part of the game and the bedrock on which everything else stands on. As such, I am expecting it will be one of (if not the longest) feature implementations in the entire process. Ideally I think it should take me about a month and a half to finish, but I anticipate it could take up to three months of development due to how broad and complex this feature is and how many considerations need to be made regarding every aspect of the game's design.


What about devlogs like this one? When can you expect more? That question is a bit more complicated... In the past I've said I would release devlogs in line with major features being finished. But I'm not sure how I feel about that idea anymore due to the large variation in the amount of time it takes to implement various features. As I recently stated, the "cosmetic" system could take up to months to finish and I don't want to leave you all hanging that long, especially with the knowledge of how many people actively follow the project. Plus, that can go the other direction too: if multiple major features are implemented quickly, I don't want to rapidly put out devlogs. So I've decided on a compromise that I think allows me to reasonably pace myself while still keeping the community happy:


You can safely expect devlogs to release at a rate of about one per month. That doesn't mean they'll come out the same day every month, it's not a strict guideline. But as a rough estimate, you can hold that expectation going forwards.


One more thing about the devlogs, and that is their length. I'm going to be honest here... writing this was absolutely brutal. I recognize part of that has to do with building an entire site to hold these logs, but even writing this post took significantly longer than I expected. It's the first time I've ever felt stressed working on this project. I only have my limited free time to work on this, and time I spend writing devlogs is time I'm not spending developing the game. So in the future you should expect devlogs to be shorter. Don't worry though, as they'll also be far more focused on the technical side of things than this one was, and I'll be sure to bring new gameplay and engine snippets with me to every single post!


Before I close this out, there's a question I'm sure a lot of you have been wondering that I want to put to rest. When will you actually get to play Everytale? I have nothing to say on the topic of a potential release date -- the game is simply too early in development for me to speak on that. But I would like to make a big commitment here. I want Everytale to, at minimum, have a publicly playable alpha build by a year from posting this devlog. I see how many people are as passionate about this concept as I am, so I would like to get a version you can play out as soon as possible! It will be a lot of work to get there, but I firmly believe it's achievable and that the effort will prove worth it.
